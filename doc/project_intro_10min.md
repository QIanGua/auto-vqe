# Agent-VQE 项目 10 分钟介绍讲稿

## 1. 开场：这个项目到底在做什么

如果我要用一句话介绍这个项目，我会说：

> Agent-VQE 是一个让 AI Agent 自动设计量子变分线路结构的实验框架，目标是更高效地逼近量子体系的基态能量。

这里先提前说明一个很容易混淆的点：

- Agent-VQE 不是 ADAPT-VQE 的别名
- ADAPT-VQE 是 Agent-VQE 这个更大框架里可以接入的一类搜索策略

传统 VQE 往往把重点放在“给定一个 Ansatz，怎么调参数”。但在真实研究里，更难的问题其实是：

- Ansatz 结构本身怎么设计
- 哪种线路更简单、却还能达到足够低的能量误差
- 当搜索空间变大后，如何避免试验过程变成不可复现的手工调参

这个项目做的事情，就是把“设计 Ansatz”这件事从研究者凭经验写死在 Python 代码里，变成一个可以被搜索、被审计、被复现的自动化过程。

也就是说，它不是单纯在跑一个 VQE，而是在构建一个“自动做 VQE 结构研究”的系统。

---

## 2. 为什么这个问题重要

VQE 是 NISQ 时代很重要的一类算法，但它一直有一个核心瓶颈：Ansatz 设计高度依赖人工经验。

如果 Ansatz 太弱：

- 表达能力不够，能量降不下来

如果 Ansatz 太强：

- 参数太多
- 线路太深
- 优化更难
- 对未来硬件也更不友好

所以我们真正想找的，不只是“最低能量”，而是“在精度和复杂度之间最合适的结构”。

这个项目把这个目标明确写成工程原则：

- 追求低能量误差
- 同时遵守奥卡姆剃刀，优先更简单的线路
- 所有实验结果都要可追踪、可对比、可复现

这意味着它不是一个一次性脚本，而是一个面向科研闭环的基础设施。

---

## 3. 项目的核心思想：把 Ansatz 搜索变成一个闭环系统

我会把这个项目的核心思想分成三层来讲。

### 第一层：把“结构”从代码中抽离出来

项目没有把线路结构硬编码成某段固定的量子线路函数，而是把它表示成结构化配置或者 `AnsatzSpec`。

比如一个配置里可以明确描述：

- 层数 `layers`
- 单比特门集合 `single_qubit_gates`
- 双比特门类型 `two_qubit_gate`
- 纠缠拓扑 `entanglement`
- 初始态 `init_state`

这样做有一个关键收益：

- 搜索算法操作的是“结构空间”
- 训练引擎操作的是“连续参数空间”

这就把“结构优化”和“参数优化”分层了。

### 第二层：把搜索算法做成可插拔策略

项目不是只支持一种搜索方式。它抽象了统一的 `SearchStrategy` 接口，然后把不同搜索方法都当作策略插件接入。

当前仓库里已经能看到几条路线：

- GA 搜索
- 多维网格搜索
- ADAPT-VQE 风格的构造式增长策略原型
- 多策略编排器 `SearchOrchestrator`

所以这个框架的野心，不是“把某个算法写出来”，而是把“自动找 Ansatz”这件事做成一个统一实验平台。

### 第三层：把实验过程做成可审计事件流

这个仓库特别有意思的一点，是它非常重视审计和复现。

每轮实验不只是打印一个最终能量，而是会系统记录：

- 配置来源
- 优化器配置
- 目标体系
- 运行环境
- Git 提交信息和工作区差异
- 结果指标
- 产物路径

最后写入 `results.jsonl` 和 Markdown 报告。

这其实是在把“科研试错过程”产品化、数据库化。

## 3.5 需要专门讲清楚的一点：它和 ADAPT-VQE 有什么区别

这一点我会在介绍里专门停半分钟讲清楚，因为很多听众一看到“自动构造 Ansatz”，会立刻联想到 ADAPT-VQE。

最简单的区别可以概括为：

- ADAPT-VQE 是一种具体算法
- Agent-VQE 是一个可容纳多种算法的框架

### ADAPT-VQE 更像什么

ADAPT-VQE 的核心思想是：

- 先定义一个算符池
- 每一轮根据梯度或某种贡献指标，挑一个最有价值的算符
- 把它追加到当前线路里
- 然后重新优化参数

它本质上是一种“贪心式、构造式”的 Ansatz 生长方法。

所以 ADAPT-VQE 的关注点通常是：

- 算符池怎么定义
- 每轮怎么选算符
- 何时停止增长

### Agent-VQE 更像什么

Agent-VQE 关注的问题更大一层。它问的是：

- 我们是否只能用 ADAPT 这一路？
- 是否也可以用 GA、Grid、未来的 BO 或 RL 来搜索结构？
- 如何让这些不同方法共用同一套评估协议和日志体系？
- 如何让搜索过程受统一预算约束，并且支持多策略切换？

所以 Agent-VQE 不是在和 ADAPT-VQE 竞争命名，而是在做“ADAPT-VQE 之上的上层框架”。

### 从代码结构上看，两者关系也很明确

- `core/adapt_vqe.py` 只是项目中的一个策略实现
- `core/strategy_base.py` 定义的是更高层的统一策略协议
- `core/controller.py` 和 `core/engine.py` 则负责把不同策略纳入同一个实验闭环

换句话说：

- ADAPT-VQE 决定“这一轮加哪个算符”
- Agent-VQE 决定“整个自动搜索系统如何组织、评估、记录和切换策略”

### 如果在汇报里要一句话区分

我会直接这样说：

> ADAPT-VQE 是一种自动长线路的方法，而 Agent-VQE 是一个统一管理多种自动长线路方法的实验框架。

### 为什么这个区别重要

因为如果把 Agent-VQE 误解成 ADAPT-VQE，就会低估这个项目真正的工程价值。

这个项目的亮点并不只是“实现了一个自动增长算法”，而是：

- 把多种 Ansatz 搜索方法统一到同一个协议下
- 把实验记录做成可审计、可复现的数据流
- 把搜索过程提升为一个可编排的研究系统

所以更准确地说：

- ADAPT-VQE 是 Agent-VQE 未来重点支持的一种策略
- 但 Agent-VQE 的目标，是成为承载 ADAPT、GA、Grid、BO、RL 等方法的通用底座

## 4. 代码架构怎么讲最清楚

如果只有 10 分钟，我会按“协议层、控制层、编译层、执行层、实验层”来讲。

### 4.1 协议层：定义大家怎么协作

`core/strategy_base.py`

- 定义搜索策略统一接口
- 让 GA、Grid、ADAPT 这类方法都能被统一调度

`core/schemas.py`

- 用结构化 schema 约束候选、评估、优化器、Ansatz 等对象
- 避免实验脚本之间的数据格式漂移

这一层的作用，是先把“搜索”变成一个有清晰边界的协议问题。

### 4.2 控制层：预算、停止规则和策略切换

`core/controller.py`

这里面最关键的两个对象是：

- `SearchController`
- `SearchOrchestrator`

`SearchController` 负责：

- 总运行次数预算
- 墙钟时间预算
- 连续无改进计数
- 连续失败计数
- 改进阈值判断

`SearchOrchestrator` 负责：

- 顺序执行多个策略
- 在无改进时触发策略切换
- 共享全局预算
- 管理候选晋级和多保真评估队列

这说明项目已经不把“搜索”看成单次调用，而是看成一个受预算约束的科研决策过程。

### 4.3 编译层：把配置翻译成可执行量子线路

`core/circuit_factory.py`

这是整个项目里非常关键的一层。它做的事情是：

- 从结构化 config 构造量子线路
- 支持不同门集和拓扑
- 支持 block/operator 级别的结构生长
- 估计参数量、双比特门数等成本

也就是说，搜索算法并不直接生成 Python 代码，而是生成结构化定义，再由 circuit factory 负责编译。

这个解耦很重要，因为它让：

- 搜索空间更容易被枚举和组合
- 后续更容易加入硬件约束
- 参数映射和 warm-start 变得可实现

### 4.4 执行层：统一 VQE 训练与日志出口

`core/engine.py`

这一层解决的是“给你一个候选结构，如何稳定地评估它”。

它负责：

- 参数优化
- VQE 训练主循环
- 实验目录管理
- 结构化 JSONL 记录
- 运行环境和 Git 信息采集
- 报告生成

我会特别强调一点：

这个项目不是只在追求“跑通”，而是在追求“评估过程标准化”。

这让不同策略出来的候选结构可以在同一套规则下比较。

### 4.5 实验层：面向具体物理体系的入口

`experiments/tfim/`
`experiments/lih/`

目前主要有两类实验对象：

- 1D TFIM
- LiH 分子

这两类任务刚好也代表两种典型场景：

- TFIM 更偏模型问题和结构搜索验证
- LiH 更偏量子化学问题和实际物理意义

项目里的 `run.py`、`ga_search.py`、`auto_search.py`、`multidim_search.py` 就构成了完整研究流程入口。

## 5. 实际工作流怎么讲

我会把项目工作流总结成一个三阶段闭环。

### 阶段一：探索

先用 GA 或 Grid 在结构空间里搜索。

输出：

- 最优配置 JSON
- 中间结果日志

### 阶段二：验证

再用统一的 `run.py` 对搜索到的最佳结构进行标准化重复评估。

这里的重点是：

- 不是只看单次幸运结果
- 要看不同 seed 下是否稳定

### 阶段三：分析与更新策略

最后读取自动生成的报告和结果表，回答几个研究问题：

- 误差降了多少
- 参数量有没有膨胀
- 双比特门代价是否过高
- 下一轮应该扩大哪一个搜索维度，还是换策略

这就是 README 里反复强调的“猜想-验证-修正”循环，也可以说是一个面向量子线路设计的科学方法闭环。

## 6. 这个项目当前最有价值的地方

如果我向一个技术团队介绍它的亮点，我会强调四点。

### 6.1 它把研究问题工程化了

很多量子算法项目停留在“单实验脚本”，而这个仓库已经明显走向“研究平台”：

- 有统一 schema
- 有统一日志
- 有统一评估协议
- 有测试护城河

### 6.2 它把“结构搜索”从“参数优化”里拆了出来

这是非常关键的抽象升级。

因为真正难扩展的是结构空间，而不是单次梯度下降。

### 6.3 它有面向更大规模系统的架构预留

虽然当前实验验证主要还是 4-qubit 级别，但从设计上已经在为更大系统准备：

- block/operator 级生长
- 参数映射
- warm-start
- 多策略编排
- 多保真评估

这说明它不是只为 demo 写的。

### 6.4 它很适合和 AI Agent 结合

这个仓库本质上很适合 agentic workflow，因为很多关键环节都已经结构化了：

- 搜索空间是结构化的
- 决策规则是显式的
- 结果日志是可机读的
- 报告是可供下一轮推理消费的

所以它特别适合作为“LLM 自动科研代理”的实验基座。

## 7. 当前进展与成熟度判断

如果我要诚实评价项目现状，我会这样讲：

它已经完成了“从脚本集合到框架骨架”的跃迁，但还处在“研究框架成型、规模化验证未完成”的阶段。

目前已经比较成熟的部分有：

- GA、Grid、Baseline 的统一框架
- `SearchController` 和 `SearchOrchestrator`
- 参数映射协议与 warm-start 相关基础设施
- 审计与日志系统
- 一套覆盖关键行为的自动化测试

还在继续补强的部分有：

- ADAPT-VQE 的进一步完善
- 更公平的多策略对比
- 面向更高比特规模的真实验证
- 更丰富的搜索后端，比如 BO 或 RL

所以它现在最像什么？

我会说，它已经是一套“值得继续堆研究资产”的底座，而不是一个一次性的实验目录。

## 10. 结尾版短讲稿

各位可以把 Agent-VQE 理解成一个“自动量子线路研究员”。  
它不是只负责调参数，而是把 Ansatz 设计本身变成搜索对象，再用统一的 VQE 引擎、预算控制器和日志系统去验证每一个结构假设。  
这个项目的意义，在于它把量子变分算法研究中最依赖经验的一步，逐步改造成一个可编排、可复现、可审计的自动化流程。  
今天它已经支持 TFIM 和 LiH 两类典型任务，也已经具备 GA、Grid、Orchestrator、参数映射和实验审计等关键基础设施。  
如果后续继续补强 ADAPT、贝叶斯优化、强化学习和大规模验证，它就有机会从一个实验项目，演进成一套真正服务于自动 Ansatz discovery 的研究平台。
